¿Qué es la cultura DevOps?

La cultura DevOps es una metodología de trabajo que promueve la colaboración entre los equipos de trabajo de desarrollo (Dev) y operaciones (Ops) con el fin de mejorar la entrega del software, mejorar la calidad y la eficiencia en los procesos. Está cultura fomenta la comunicación, integración y automatización de procesos para mejorar el desarrollo del software. Se enfoca en principios claves tales como:

  • - Automatización
  • - Colaboración
  • - Monitoreo continuo
  • - Mejora continua

¿Qué es CI/CD?

Es un conjunto de prácticas y principios en el desarrollo de software que busca como propósito automatizar y optimizar el proceso de entrega de aplicaciones. CI significa Integración Continua mientras que CD significa Despliegue Continuo. Está práctica es importante dentro del marco de DevOps. Pero que significa cada una?
CI: Hace referencia a la práctica de integrar cambios de código en un repositorio compartido varias veces en el día. Además cada vez que se realiza un cambio se ejecuta una prueba automatizada para asegurar que el nuevo codigo no contenga errores. Todo esto permite detectar problemas en etapas tempranas permitiendo que su solución sea más fácil y no permite que se convierta en una falla mayor.

¿Qué es un Pipeline CI/CD?

Un Pipeline CI/CD es un conjunto de pasos automáticos que ayuda en el desarrollo, prueba y despliegue de aplicaciones de software de manera continua. CI, que significa Integración Continua, permite a los programadores unir sus cambios en el código en un repositorio compartido varias veces al día. Cada vez que se hace una integración, se realizan pruebas automáticas para detectar y solucionar errores rápidamente.
Por otro lado, CD puede significar Entrega Continua o Despliegue Continuo. En el caso de Entrega Continua, el código se prepara para ser lanzado a producción automáticamente, pero necesita que alguien lo apruebe antes de ser publicado. En cambio, el Despliegue Continuo automatiza todo el proceso, así que cualquier cambio que supere las pruebas se envía a producción sin que nadie tenga que intervenir.

¿Qué es Git y Github?

Git es una herramienta que permite a los desarrolladores llevar un registro de los cambios que hacen en el código y trabajar juntos en proyectos de software. Ayuda a gestionar diferentes versiones del código y facilita que varias personas colaboren al mismo tiempo en el mismo proyecto.
Mientras que GitHub es una plataforma en línea que utiliza Git para almacenar y gestionar repositorios de código. Ofrece funciones para colaborar, como hacer *pull requests*, gestionar problemas y crear documentación, lo que hace que trabajar en equipo y seguir el progreso de los proyectos sea más sencillo. En resumen, Git es el sistema de control de versiones, mientras que GitHub es el lugar donde se puede utilizar Git para colaborar con otros.

¿Qué son GitHub Actions?

GitHub Actions es una herramienta de automatización integrada en GitHub que permite crear flujos de trabajo personalizados para gestionar el ciclo de vida de un proyecto. Los flujos de trabajo se definen en archivos YAML y se pueden configurar para ejecutarse en respuesta a diversos eventos en el repositorio, como push, pull requests o la creación de versiones.
Los componentes principales de GitHub Actions son:
1. Eventos: Son los disparadores que inician un flujo de trabajo. Pueden ser acciones en el repositorio, como un push de código, la creación de un pull request, o incluso eventos programados. Estos eventos permiten que el flujo de trabajo se ejecute automáticamente cuando ocurren ciertas condiciones.
2. Jobs: Un flujo de trabajo está compuesto por uno o más trabajos, que son conjuntos de pasos que se ejecutan en un entorno específico. Cada trabajo se ejecuta en un contenedor o máquina virtual y puede ejecutarse de forma independiente o en paralelo con otros trabajos en el mismo flujo de trabajo.
3. Steps: Son las acciones individuales dentro de un trabajo. Cada paso puede ejecutar comandos, ejecutar scripts, o llamar a otras acciones. Los pasos se ejecutan en el orden en que están definidos y permiten realizar tareas específicas como compilar código, ejecutar pruebas o desplegar aplicaciones.
